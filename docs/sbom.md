# SBOM generation

`argus sbom [path]` generates a Software Bill of Materials: the inventory of
components and dependencies in a target, in a standard interchange format
(CycloneDX or SPDX). It is increasingly required by procurement and executive
orders, and it is cheap on top of the dependency inventory Argus already
builds for the [SCA pass](coverage.md), so the SBOM and the vulnerability
report describe the same set of components.

```bash
argus sbom .                                  # CycloneDX (JSON) to stdout
argus sbom . --format spdx-json -o sbom.json  # SPDX (JSON) to a file
argus sbom ./service -o service.cdx.json      # scope to a subtree
```

## Formats

| `--format` | Standard | Encoding |
|---|---|---|
| `cyclonedx` (default) | OWASP CycloneDX | JSON |
| `spdx-json` | Linux Foundation SPDX | JSON |
| `spdx` | Linux Foundation SPDX | tag-value (text) |

CycloneDX JSON is the default because it is what most procurement and
supply-chain tooling asks for; SPDX is offered for consumers standardized on
it.

## An SBOM is an artifact, not findings

An SBOM lists what is **present**, not what is **wrong**. So `argus sbom`
produces a document, never findings, and never gates a build. To gate on the
*vulnerabilities* in those components, use `argus scan` (the SCA pass), which
inventories the same dependencies and reports the ones with known CVEs. The
two are complementary: the SBOM is the manifest you hand to a customer or an
auditor; the scan is the gate you run in CI.

## Faithful output

Argus generates the SBOM with trivy, the same engine as the SCA pass, and
returns its output **unmodified**: no re-serialization, no field rewriting.
The bytes Argus emits are exactly what the SBOM engine produced, so the
document validates against the upstream CycloneDX or SPDX schema without any
argus-specific quirks. (The `serialNumber` and `bom-ref` identifiers are
freshly generated by the engine on each run, so two runs of the same target
differ only in those identifiers, as the specs intend.)

Install trivy from [its releases](https://github.com/aquasecurity/trivy) or
your package manager; `argus sbom` reports honestly if it is not on `PATH`.

## From the console

The Operate tab has a **Download SBOM** control (with a format selector) for
the served repo and any directory or git target: it generates the document on
the server and downloads it in the browser. Cloud, DAST, and image targets
have no component tree, so the control is offered only for filesystem targets.
